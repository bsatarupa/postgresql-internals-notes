# PostgreSQL Visibility Map, Freezing & VACUUM

## Why does PostgreSQL Freeze Tuples?

Every tuple stores:

```text
xmin = transaction that inserted the tuple
xmax = transaction that deleted/updated the tuple
```

Example:

```text
xmin = 100
xmax = 0
```

Transaction IDs (XIDs) are **32-bit**, so they eventually wrap around.

```text
4294967295
      ↓
      0
      ↓
      1
```

Without freezing, PostgreSQL could confuse an ancient transaction with a new one.

To prevent this, VACUUM replaces an old committed `xmin` with **FrozenXID**.

Before:

```text
xmin = 100
```

After:

```text
xmin = FrozenXID
```

> **FrozenXID means:** "This tuple is so old that it is permanently considered committed."

---

# Visibility Map (VM)

The Visibility Map is **not maintained per tuple**.

It is maintained **per heap page**.

Each heap page has two bits:

```text
ALL_VISIBLE
ALL_FROZEN
```

---

# ALL_VISIBLE

Meaning:

> Every live tuple on this page is visible to every transaction.

Benefits:

- VACUUM can skip the page.
- Enables Index-Only Scan.

Example:

```text
Heap Page

Alice
Bob
Charlie

↓

ALL_VISIBLE = 1
```

---

# ALL_FROZEN

Meaning:

> Every tuple on this page has already been frozen.

Example:

```text
xmin = FrozenXID
xmin = FrozenXID
xmin = FrozenXID

↓

ALL_FROZEN = 1
```

These pages are safe from transaction ID wraparound.

---

# Relationship

```
              ALL_VISIBLE
             /           \
            /             \
           /               \
     ALL_FROZEN       Visible but
                       not frozen
```

Or

```
ALL_FROZEN ⊂ ALL_VISIBLE
```

Meaning:

✅ Every ALL_FROZEN page is ALL_VISIBLE.

❌ Every ALL_VISIBLE page is **not** necessarily ALL_FROZEN.

Possible:

```text
ALL_VISIBLE = 1
ALL_FROZEN = 0
```

Impossible:

```text
ALL_VISIBLE = 0
ALL_FROZEN = 1
```

---

# When is ALL_VISIBLE Set?

Only **VACUUM** sets it.

VACUUM verifies:

- no dead tuples remain
- every remaining tuple is visible to every transaction

Then:

```text
ALL_VISIBLE = 1
```

---

# When is ALL_VISIBLE Cleared?

Immediately when a page changes.

Examples:

```
INSERT
UPDATE
DELETE
```

Timeline:

```
INSERT

↓

ALL_VISIBLE = 0

↓

COMMIT

↓

Still 0

↓

VACUUM

↓

ALL_VISIBLE = 1

↓

UPDATE

↓

ALL_VISIBLE = 0

↓

VACUUM

↓

ALL_VISIBLE = 1
```

> **Commit never sets ALL_VISIBLE. Only VACUUM does.**

---

# Does VACUUM Scan Every Page?

No.

VACUUM first consults the Visibility Map.

```
Page 1
ALL_VISIBLE = 1

↓

Skip

-------------------

Page 2
ALL_VISIBLE = 0

↓

Read page

↓

Scan every tuple

-------------------

Page 3
ALL_VISIBLE = 1

↓

Skip
```

This is why VACUUM scales well for large tables.

---

# If VACUUM Reads a Page...

It scans every tuple.

Conceptually:

```cpp
for each tuple
{
    check visibility
    check dead status
    check freeze eligibility
    check removal eligibility
}
```

---

# How Does VACUUM Decide a Tuple Can Be Removed?

Example:

```
xmin = FrozenXID
xmax = 500
```

Suppose

```
OldestXmin = 1000
```

Since

```
500 < OldestXmin
```

every active transaction started after the delete.

Therefore:

```
No snapshot can ever see this tuple.
```

VACUUM removes it.

---

# Does Freezing Prevent Deletion?

No.

Freezing changes only **xmin**.

Before delete:

```
xmin = FrozenXID
xmax = 0
```

After delete:

```
xmin = FrozenXID
xmax = 900
```

VACUUM still removes it once the delete is invisible to every active transaction.

Deletion depends mainly on:

- `xmax`
- active snapshots (`OldestXmin`)

Not on whether the tuple is frozen.

---

# Index-Only Scan

Normally PostgreSQL does:

```
Index

↓

Heap

↓

Check MVCC visibility
```

because indexes do **not** store visibility information.

If

```
ALL_VISIBLE = 1
```

PostgreSQL already knows every tuple is visible.

So it can do:

```
Index

↓

Return row
```

without reading the heap.

---

# Freezing vs Visibility Map

| Freezing | Visibility Map |
|----------|----------------|
| Per tuple | Per page |
| Replaces old `xmin` with `FrozenXID` | Stores page summary |
| Prevents XID wraparound | Helps VACUUM |
| Done by VACUUM | Updated by VACUUM (set) and DML (clear) |

---

# Mental Model

Imagine a library.

Each heap page is a bookshelf.

The Visibility Map is a sticky note attached to the shelf.

🟢 **Green Sticker (ALL_VISIBLE)**

```
Every book on this shelf is available to everyone.
```

🔵 **Blue Sticker (ALL_FROZEN)**

```
Every book has already been permanently archived.
```

Each individual book (tuple) still has its own labels:

```
xmin
xmax
```

Freezing changes only the tuple.

The Visibility Map summarizes the entire page.

# Quick Revision

✅ `xmin` = inserting transaction

✅ `xmax` = deleting/updating transaction

✅ VACUUM freezes old `xmin`

✅ Frozen tuple can still be deleted later

✅ Visibility Map is maintained per page

✅ `ALL_VISIBLE` ⇒ every tuple is visible to everyone

✅ `ALL_FROZEN` ⇒ every tuple is frozen

✅ `ALL_FROZEN` ⊂ `ALL_VISIBLE`

✅ INSERT / UPDATE / DELETE clear `ALL_VISIBLE`

✅ Only VACUUM sets `ALL_VISIBLE`

✅ VACUUM skips `ALL_VISIBLE` pages

✅ If a page is scanned, every tuple is examined

✅ Tuple removal depends on `xmax` and active snapshots

✅ Visibility Map enables efficient VACUUM and Index-Only Scans
